# Partner Key Bot

Универсальный шаблон Telegram-бота с мини-приложением, локальным хранилищем и опциональными AI-интеграциями. Проект можно адаптировать под разные пользовательские сценарии: консультационный бот, персональный ассистент, продуктовый onboarding, генератор рекомендаций или закрытый сервис для ограниченной аудитории.

README намеренно описывает устойчивую архитектуру и правила эксплуатации, а не конкретный сценарий диалога: продуктовая логика может меняться без необходимости переписывать базовую документацию.

## Постоянные возможности

- Telegram-бот на `python-telegram-bot` с long polling запуском.
- Telegram Web App для сбора, показа или сохранения пользовательских данных.
- Локальное SQLite-хранилище для пользователей, профилей, истории и продуктовых состояний.
- Конфигурация через переменные окружения и `.env` для локального запуска.
- Поддержка публичного режима и режима доступа по списку Telegram ID.
- Опциональная интеграция с OpenAI API для генерации, улучшения или персонализации текстов.
- Готовность к деплою в Docker/Railway с постоянным хранилищем через volume.
- Healthcheck endpoint для проверки доступности процесса.

## Архитектура проекта

```text
app/
  config.py              # настройки окружения и runtime-валидация
  storage.py             # SQLite-хранилище и работа с данными
  webapp.py              # Telegram Web App и HTTP endpoints
  services/              # внешние сервисы и AI-клиенты
  ui/                    # клавиатуры и элементы Telegram-интерфейса
  astro/                 # доменная логика текущего сценария
  woman_flow.py          # основной production entrypoint
```

Доменная логика может быть заменена или расширена без изменения инфраструктурных частей: конфигурации, хранения данных, Telegram-обвязки, Web App и деплоя.

## Быстрый старт

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
python -m app.woman_flow
```

Минимально нужен `TELEGRAM_BOT_TOKEN` в `.env`.

## Переменные окружения

| Переменная | Обязательна | Описание |
| --- | --- | --- |
| `TELEGRAM_BOT_TOKEN` | да | Токен Telegram-бота. |
| `AUTHORIZED_TELEGRAM_IDS` | нет | Список разрешённых Telegram ID через запятую. Пусто = публичный доступ. |
| `BROADCAST_ADMIN_IDS` | нет | Админы служебных рассылок через запятую. |
| `APP_TIMEZONE` | нет | Таймзона приложения, по умолчанию `Europe/Moscow`. |
| `DATA_DIR` | нет | Папка для SQLite-файла. Если не задана, на Railway автоматически используется `RAILWAY_VOLUME_MOUNT_PATH`, иначе `data`. |
| `WEBAPP_URL` | нет | Публичный URL Telegram Web App. Если путь не указан, добавляется `/webapp`. |
| `OPENAI_API_KEY` | нет | Ключ OpenAI для AI-функций. Базовый сценарий должен работать без него. |
| `OPENAI_MODEL` | нет | Модель OpenAI, по умолчанию `gpt-4.1-mini`. |

## Данные и хранение

Приложение хранит состояние в SQLite-файле внутри `DATA_DIR`. Путь выбирается в таком порядке:

1. `DATA_DIR`, если переменная задана явно;
2. `RAILWAY_VOLUME_MOUNT_PATH`, если Railway передал путь volume;
3. `/data` на Railway;
4. локально — `data`.

Для production обязательно подключите постоянный volume. Если volume не подключён к выбранному пути, SQLite-файл останется внутри контейнера и может быть потерян при redeploy или restart.

## Production

Railway запускает основной сценарий командой:

```bash
python -m app.woman_flow
```

Внутри процесса поднимается HTTP-сервер на `PORT` и отдаёт постоянные endpoints:

- `/healthz` — healthcheck;
- `/webapp` — Telegram Web App;
- `/api/profile` — API профиля пользователя.

Dockerfile содержит базовый `CMD`, но при деплое на Railway используется `startCommand` из `railway.json`.

## Как безопасно менять продуктовую логику

- Оставляйте инфраструктурные контракты стабильными: переменные окружения, entrypoint, storage path и healthcheck.
- Новую бизнес-логику выносите в отдельные модули, чтобы не смешивать её с конфигурацией и хранением.
- AI-функции делайте опциональными: отсутствие `OPENAI_API_KEY` не должно ломать базовый сценарий.
- Для пользовательских воронок фиксируйте ключевые события: старт, переход в Web App, сохранение профиля, генерация результата, ошибка, возврат пользователя.
- При изменении сценария обновляйте тексты интерфейса и продуктовые метрики, но не превращайте README в описание конкретной временной механики.
